纠正：
1.@Slf4j 会在编译阶段自动生成这行代码
private static final org.slf4j.Logger log =
        org.slf4j.LoggerFactory.getLogger(LoginInterceptor.class);
所以你代码里才能直接写：
log.warn("JWT 过期: {}", e.getMessage());
而不会报错。
为什么 IDE 里看不见？
因为 Lombok 是 编译期生成代码：
源码
  ↓
Lombok 处理
  ↓
生成完整代码
  ↓
编译

IDE 默认只显示你写的源码。

2.Bean 的作用是：
让 Spring 管理这个对象。
好处是：
1️⃣ 可以自动依赖注入
2️⃣ 生命周期由 Spring 管理
3️⃣ 可以被框架自动使用

3.@RequiredArgsConstructor真正作用是：Lombok 自动生成构造方法。
例如：
private final JwtUtil jwtUtil;
Lombok 会自动生成：
public LoginInterceptor(JwtUtil jwtUtil) {
    this.jwtUtil = jwtUtil;
}
然后 Spring 用 构造器注入。


4.AUTH_HEADER 这里
你写：
这个是Jason中数据头 
❗这里是错误的。
不是 JSON。
这是：
HTTP 请求头
完整请求一般是：
GET /api/books
Authorization: Bearer eyJhbGciOiJIUzI1Ni...
所以：
request.getHeader("Authorization")
是从 HTTP Header 里拿 token。

5.BEARER_PREFIX
你写：
规定数据格式 
这个说法可以，但更准确一点：
JWT 标准认证格式
Authorization: Bearer token
为什么要加 Bearer？
因为 HTTP 认证规范规定：
Bearer token
表示：
携带令牌认证

6.preHandle
你写：
拦截器定义 
这里建议你写清楚：
preHandle()
拦截器执行方法
Controller 执行之前会先执行这里
流程：
请求 -> 拦截器 -> Controller

7.setAttribute 这一句
你写：
这三行作用不大，感觉不如用ThreadLocal 
这里我给你一个 很重要的纠正。
你的理解有点偏。
这三行作用 非常大：
request.setAttribute("currentUserId", userId);
Controller 里可以直接拿：
request.getAttribute("currentUserId")
这是 SpringMVC最标准的做法。
ThreadLocal 什么时候用？
ThreadLocal 通常用于：
MyBatis
用户上下文
全局用户信息
例如：
UserContext.getCurrentUser()
很多框架会这么写。
但现在这个写法：
request.setAttribute()
完全合理，而且更简单
ThreadLocal 这条 不一定更好。
原因：
ThreadLocal 需要手动清理
否则可能内存泄漏
而 request.setAttribute：
请求结束自动释放
所以当前写法其实很好。
